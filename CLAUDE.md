## What This Repo Is

Infrastructure-as-Code and GitOps configuration for a single production Talos Kubernetes cluster (`talos-prd`) running on Proxmox VE. All cluster state is managed through ArgoCD after a one-time Terraform bootstrap.

## Common Commands

### Bootstrap (one-time cluster setup)
```bash
# Load secrets from 1Password before running Terraform
eval $(op signin)
source resources/bootstrap/scripts/load-secrets.sh

# Deploy Proxmox VMs + Talos cluster
cd resources/bootstrap && tofu init && tofu apply

# Then bootstrap ArgoCD GitOps stack
cd resources/gitops-config
./scripts/generate-input-files.sh   # pull secrets from 1Password into input-files/
source ./scripts/load-secrets.sh
tofu init && tofu apply
```

### Talos operations
```bash
talosctl --talosconfig resources/bootstrap/output/talosconfig <subcommand>
kubectl --kubeconfig resources/bootstrap/output/kubeconfig <subcommand>

# Rolling Talos upgrade
resources/bootstrap/scripts/upgrade-talos.sh
```

### Validate Terraform
```bash
cd resources/bootstrap && tofu validate
cd resources/gitops-config && tofu validate
```

## Architecture

### Two-Phase Bootstrap
1. **Terraform** (one-time): Creates VMs, bootstraps Talos, installs 1Password Connect + ESO + ArgoCD via Helm, deploys the root `sync-app`.
2. **ArgoCD takes over**: `sync-app` (App-of-Apps) manages everything via GitOps — including re-managing 1Password Connect and ESO, replacing their Terraform-deployed instances.

### Secrets
Secrets never live in git. The flow is: **1Password vault → 1Password Connect → External Secrets Operator → Kubernetes Secrets**. Apps reference `ExternalSecret` CRDs that pull from 1Password at runtime.

The `input-files/` directory holds bootstrap-time secret YAML files (gitignored) generated by `scripts/generate-input-files.sh`.

### Sync Waves (ArgoCD deployment order)
- Wave 0: Longhorn, Cilium, ArgoCD (self-managed)
- Wave 1: 1Password Connect, External Secrets Operator, cert-manager, external-dns, kubelet-csr-approver, cloudnative-pg
- Wave 2: kube-prometheus-stack, Loki, Promtail, Authentik, prometheus-pve-exporter
- Wave 3: proxmox (in-cluster infra manifests)
- Wave 4: home-assistant, emqx, zigbee2mqtt
- Wave 5: evcc

### App/Operator Pattern
Each operator and application is a local Helm chart deployed by an ArgoCD `Application` resource in `sync-app/templates/`. Structure:
- `Chart.yaml` — chart metadata
- `values.yaml` — all configuration
- `templates/` — Kubernetes manifests (Deployments, PVCs, HTTPRoutes, ExternalSecrets, etc.)

### Networking
Cilium with Gateway API. Two gateways:
- `app-gateway` (10.0.10.241) — user-facing apps (HTTPS)
- `svc-gateway` (10.0.10.240) — internal/operator UIs (ArgoCD, Grafana, Longhorn, Authentik admin)

Ingress uses `HTTPRoute` resources. No traditional Ingress objects.

#### CiliumNetworkPolicy patterns (cluster-specific gotchas)

**Port = container port, not service port.** Cilium enforces at the pod after DNAT. If a service maps `80 → 8080`, the policy must allow port `8080`.

**Cross-namespace `fromEndpoints` requires the namespace label.** Without it, Cilium does not reliably match across namespaces:
```yaml
- fromEndpoints:
  - matchLabels:
      k8s:io.kubernetes.pod.namespace: monitoring   # required
      app.kubernetes.io/name: prometheus
```

**kube-apiserver webhook ingress: use `cluster`, not `kube-apiserver`.** In Talos, the API-server is a static pod in the host network. Cilium does not recognise it as the `kube-apiserver` entity for *ingress* traffic (admission webhooks). Use `fromEntities: cluster` for webhook ports. `toEntities: kube-apiserver` for *egress* works fine.

**Admission webhook deadlock.** If a NetworkPolicy blocks the webhook that validates the CRD being synced by ArgoCD, the sync will loop forever and never apply the fix. Break it with a direct `kubectl patch` on the CiliumNetworkPolicy, then re-trigger the ArgoCD sync.

**Standard allow rules (template for every namespace):**
```yaml
# DNS egress — every namespace needs this
- toEndpoints:
  - matchLabels:
      k8s:io.kubernetes.pod.namespace: kube-system
      k8s-app: kube-dns
  toPorts:
  - ports:
    - port: "53"
      protocol: UDP
    - port: "53"
      protocol: TCP

# Prometheus scraping ingress
- fromEndpoints:
  - matchLabels:
      k8s:io.kubernetes.pod.namespace: monitoring
      app.kubernetes.io/name: prometheus
  toPorts:
  - ports:
    - port: "<metrics-port>"
      protocol: TCP

# kube-apiserver egress (controllers/operators only)
- toEntities:
  - kube-apiserver
  toPorts:
  - ports:
    - port: "6443"
      protocol: TCP
```

### Storage
Longhorn with four storage classes:
- `longhorn-fast`: 3 replicas (databases, critical data)
- `longhorn-standard`: 2 replicas (general app data) — default class
- `longhorn-archive`: 1 replica, Delete reclaim (backups, low-priority)
- `longhorn-monitoring`: 1 replica, strict-local (metrics/log data, no backups)

S3 backups go to Hetzner Object Storage.

### Authentication
Authentik provides OIDC/SSO for all apps that support it. Apps that need OIDC reference Authentik as their identity provider.

### Database
CloudNative-PG operator provides shared PostgreSQL. Apps connect to `cnpg-shared-rw.cnpg-system.svc`.

### Dependency Updates
Renovate runs Monday before 6am and opens PRs. Tier 0/1 (infra/core operators) get individual PRs; Tier 2+ app patch updates are auto-merged.

## Security Audit

`security/` (gitignored) contains the live security audit for this cluster:
- `findings.md` — alle bevindingen met status (opgelost / gedeeltelijk / open)
- `rules.md` — werkregels voor het oplossen van findings (functionality first, one commit per fix, etc.)
- `namespace-status.md` — overzicht van PSS labels en CiliumNetworkPolicy per namespace

Before adding a new NetworkPolicy or PSS labels, check `namespace-status.md` for current state.

## Terraform State

Remote state is stored in Hetzner Object Storage (S3-compatible) at `nbg1.your-objectstorage.com`. Credentials are loaded via the `load-secrets.sh` scripts before running `tofu`.
