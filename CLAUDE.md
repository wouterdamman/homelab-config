## What This Repo Is

Infrastructure-as-Code and GitOps configuration for a single production Talos Kubernetes cluster (`talos-prd`) running on Proxmox VE. All cluster state is managed through ArgoCD after a one-time Terraform bootstrap.

## Common Commands

### Bootstrap (one-time cluster setup)
```bash
# Load secrets from 1Password before running Terraform
eval $(op signin)
source resources/bootstrap/scripts/load-secrets.sh

# Deploy Proxmox VMs + Talos cluster
cd resources/bootstrap && tofu init && tofu apply

# Then bootstrap ArgoCD GitOps stack
cd resources/gitops-config
./scripts/generate-input-files.sh   # pull secrets from 1Password into input-files/
source ./scripts/load-secrets.sh
tofu init && tofu apply
```

### Talos operations
```bash
talosctl --talosconfig resources/bootstrap/output/talosconfig <subcommand>
kubectl --kubeconfig resources/bootstrap/output/kubeconfig <subcommand>

# Rolling Talos upgrade
resources/bootstrap/scripts/upgrade-talos.sh
```

### Validate Terraform
```bash
cd resources/bootstrap && tofu validate
cd resources/gitops-config && tofu validate
```

## Architecture

### Two-Phase Bootstrap
1. **Terraform** (one-time): Creates VMs, bootstraps Talos, installs 1Password Connect + ESO + ArgoCD via Helm, deploys the root `sync-app`.
2. **ArgoCD takes over**: `sync-app` (App-of-Apps) manages everything via GitOps — including re-managing 1Password Connect and ESO, replacing their Terraform-deployed instances.

### Secrets
Secrets never live in git. The flow is: **1Password vault → 1Password Connect → External Secrets Operator → Kubernetes Secrets**. Apps reference `ExternalSecret` CRDs that pull from 1Password at runtime.

The `input-files/` directory holds bootstrap-time secret YAML files (gitignored) generated by `scripts/generate-input-files.sh`.

### Sync Waves (ArgoCD deployment order)
- Wave 0: Longhorn, Cilium, ArgoCD (self-managed)
- Wave 1: 1Password Connect
- Wave 2: External Secrets Operator
- Wave 3: cert-manager
- Wave 5+: Applications and remaining operators

### App/Operator Pattern
Each operator and application is a local Helm chart deployed by an ArgoCD `Application` resource in `sync-app/templates/`. Structure:
- `Chart.yaml` — chart metadata
- `values.yaml` — all configuration
- `templates/` — Kubernetes manifests (Deployments, PVCs, HTTPRoutes, ExternalSecrets, etc.)

### Networking
Cilium with Gateway API. Ingress uses `HTTPRoute` resources pointing to the `app-gateway` Gateway. No traditional Ingress objects.

### Storage
Longhorn with three storage classes:
- `longhorn-fast`: 3 replicas (databases, critical data)
- `longhorn-standard`: 2 replicas (general app data)
- `longhorn-archive`: 1 replica (backups, low-priority)

S3 backups go to Hetzner Object Storage.

### Authentication
Authentik provides OIDC/SSO for all apps that support it. Apps that need OIDC reference Authentik as their identity provider.

### Database
CloudNative-PG operator provides shared PostgreSQL. Apps connect to `cnpg-shared-rw.cnpg-system.svc`.

### Dependency Updates
Renovate runs Monday before 6am and opens PRs. Tier 0/1 (infra/core operators) get individual PRs; Tier 2+ app patch updates are auto-merged.

## Terraform State

Remote state is stored in Hetzner Object Storage (S3-compatible) at `nbg1.your-objectstorage.com`. Credentials are loaded via the `load-secrets.sh` scripts before running `tofu`.
